트러블슈팅

기능 구현 과정에서 마주친 문제를 원인 분석부터 해결 결과까지 정리했습니다.

문제 상황: 쇼핑몰 프로젝트에서 장바구니 → 주문서 → 결제 단계로 이동할 때 세션 상태가 어긋나 주문 정보가 비정상 저장되는 경우가 있었습니다.
원인/어려움: 로그인 검증, 장바구니 데이터 갱신, 관리자 주문 상태 처리 로직이 각 페이지에 분산되어 있어 동일 요청이 중복 반영되었습니다.
해결 방법: 주문 처리 진입 지점을 하나로 고정하고, 세션 체크·권한 분기·재고/주문 업데이트 순서를 공통 로직으로 정리해 단계별 검증 후 저장하도록 변경했습니다.
결과/배운 점: 주문 흐름이 안정화되어 테스트 중 중복 주문 케이스를 줄였고, 상태 관리는 기능 단위보다 “요청 흐름 단위”로 설계해야 한다는 기준을 세웠습니다.

쇼핑몰 프로젝트

관련 기술: PHP · Bootstrap 문제 유형: 주문 상태 관리 · 세션 흐름
문제 상황: e스포츠 프로젝트에서 경기 목록/상세를 Ajax로 갱신할 때 필터를 변경하면 화면이 간헐적으로 전체 새로고침되어 검색 맥락이 끊겼습니다.
원인/어려움: 라우팅 응답이 HTML 렌더링과 JSON 반환을 혼용하고 있었고, 권한 체크 실패 시 리다이렉트가 발생해 프론트에서 예외 흐름을 구분하기 어려웠습니다.
해결 방법: 조회 API를 JSON 전용으로 분리하고, Laravel 라우트에서 권한 실패 응답 코드를 명확히 나눈 뒤 프론트에서 상태별 렌더링과 에러 메시지 처리를 분기했습니다.
결과/배운 점: 불필요한 전체 새로고침 없이 필요한 영역만 갱신되도록 개선했고, 화면 성능보다 먼저 “응답 형식 일관성”을 맞춰야 디버깅이 빨라진다는 점을 확인했습니다.
ES

e스포츠 프로젝트

관련 기술: Laravel · MySQL 문제 유형: Ajax 갱신 · 라우팅 응답 분리
문제 상황: 베이커리 판매관리 프로젝트에서 상품 상세/재고 화면 연동 시 특정 조건에서 데이터가 비어 보이거나 예외가 발생했습니다.
원인/어려움: 재고, 판매, 상품 테이블의 조인 키 기준이 화면별로 달라 null 데이터가 전달됐고, Git 병합 과정에서 쿼리 파일 경로가 달라져 배포 환경에서만 오류가 재현되었습니다.
해결 방법: 테이블 관계와 조인 기준을 단일 규칙으로 맞추고, 상세 조회 쿼리에 예외 처리(기본값/없는 데이터 안내)를 추가했습니다. 배포 스크립트와 경로 규칙도 함께 정리했습니다.
결과/배운 점: 상세 페이지 안정성이 높아졌고, DB 예외 처리와 Git 배포 규칙을 함께 관리해야 운영 단계의 장애를 줄일 수 있다는 점을 프로젝트에서 체득했습니다.
